HOW TO USE THIS GUIDE
─────────────────────
• Use each service's Swagger UI directly (ports 8081-8086) — pick the server
  that matches the service under test (e.g. http://localhost:8081 for IAM).
• In Swagger UI: click "Authorize" → paste just the token (no "Bearer " prefix).
  Swagger adds "Bearer " automatically.
• Replace <ADMIN_TOKEN>, <SM_TOKEN>, etc. with the actual JWT you receive
  from the login/register response.
• IDs (userId, serviceId, slaId, etc.) are assigned by the DB starting at 1.
  Adjust if your DB already has existing rows.
• Tests are numbered and ordered — run them sequentially.
• "PREDICTED OUTPUT" shows the HTTP status and the key fields to verify.
• X-User-Id / X-User-Email / X-User-Role headers are injected automatically
  by the JWT filter — you will NOT see them as input fields in Swagger UI.

WORKFLOW DIAGRAM CORRECTION
══════════════════════════════════════════════════════════════════════════

The workflow diagram provided is CORRECT with the following two notes:

1. STAGE 4 — SLA metric field changed:
   The 'metric' field in POST /api/sla now accepts the MetricType ENUM, not
   a free-text string. Valid values are:
     CALL_DURATION, DATA_TRANSFER, MESSAGE_COUNT,
     VIDEO_MINUTES, API_CALLS, BANDWIDTH
   Using any other string now returns 400 Bad Request at request validation.

2. STAGE 12 — Gateway header flow updated:
   The flow now shows:
     Gateway → injects X-User-Email, X-User-Role, X-User-Id headers
     Microservice JwtAuthFilter → Path 1 (gateway headers): reads all 3 headers
     Microservice JwtAuthFilter → Path 2 (Swagger/direct): extracts all 3
       values from JWT and wraps the request so controllers can read them.
   Previously Path 2 was missing the header injection, causing 400 errors
   when testing services directly via Swagger.
