Open MySQL command line or MySQL Workbench and run:

  mysql -u root -p

Then inside MySQL prompt:

  -- OR run files individually in this order:
  SOURCE C:/Users/.../database/01_iam_db.sql;
  SOURCE C:/Users/.../database/02_provisioning_db.sql;
  SOURCE C:/Users/.../database/03_usage_db.sql;
  SOURCE C:/Users/.../database/04_incident_db.sql;
  SOURCE C:/Users/.../database/05_analytics_db.sql;
  SOURCE C:/Users/.../database/06_notification_db.sql;

Verify databases exist:
  SHOW DATABASES;
  -- Should show: iam_db, provisioning_db, usage_db,
  --              incident_db, analytics_db, notification_db

Check application.yml to confirm. If it uses usage_sla_db instead:
  CREATE DATABASE IF NOT EXISTS usage_sla_db;

SECTION 2 — SERVICE STARTUP ORDER
══════════════════════════════════════════════════════════════════

Start each service in its own terminal window. Wait for each to
say "Started ... in X.X seconds" before starting the next.

STEP 1: MySQL must be running (Windows Services or XAMPP/WAMP)

STEP 2: Start Eureka Server
  cd comnet360/eureka-server
  mvn spring-boot:run
  Wait for: "Started EurekaServerApplication"
  Verify: http://localhost:8761 shows Eureka dashboard

STEP 3: Start API Gateway
  cd comnet360/gateway
  mvn spring-boot:run
  Wait for: "Started GatewayApplication"

STEP 4: Start IAM Service
  cd comnet360/services/iam-service
  mvn spring-boot:run
  Wait for: "✅ Admin password re-synced on startup"
  or: "✅ Default admin created on startup"

STEP 5: Start Provisioning Service
  cd comnet360/services/provisioning-service
  mvn spring-boot:run

STEP 6: Start Usage/SLA Service
  cd comnet360/services/usage-sla-service
  mvn spring-boot:run

STEP 7: Start Incident Service
  cd comnet360/services/incident-service
  mvn spring-boot:run

STEP 8: Start Analytics Service
  cd comnet360/services/analytics-service
  mvn spring-boot:run

STEP 9: Start Notification Service
  cd comnet360/services/notification-service
  mvn spring-boot:run

STEP 10: Start Frontend
  cd comnet360-frontend
  npm install   (first time only)
  npm run dev
  Open: http://localhost:5173

SECTION 3 — BACKEND API TESTING (cURL COMMANDS)
══════════════════════════════════════════════════════════════════

All cURL commands go through the API Gateway on port 8001.
The gateway adds X-User-Id, X-User-Email, X-User-Role headers
automatically based on the JWT token.

────────────────────────────────────────────────
3.1  IAM SERVICE — Authentication
────────────────────────────────────────────────

STEP 1: LOGIN AS ADMIN
──────────────────────
curl -X POST http://localhost:8001/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{
    "email": "admin@comnet360.com",
    "password": "Admin@123"
  }'

Expected Response (200 OK):
{
  "accessToken": "eyJhbGciOiJIUzUxMiJ9...",
  "refreshToken": "eyJhbGciOiJIUzUxMiJ9...",
  "tokenType": "Bearer",
  "expiresIn": 86400000,
  "userId": 1,
  "name": "System Admin",
  "email": "admin@comnet360.com",
  "role": "ADMIN"
}

SAVE THE ACCESS TOKEN — you will use it in all subsequent calls.
Replace <ADMIN_TOKEN> with your actual token below.

  Incident     : http://localhost:8084/swagger-ui.html
  Analytics    : http://localhost:8085/swagger-ui.html
  Notification : http://localhost:8086/swagger-ui.html

NOTE: When calling through Swagger directly (not gateway), the gateway
does NOT inject X-User-Id, X-User-Email, X-User-Role headers.
Some endpoints accept these as optional and fall back to 0/null.
For full functionality, always test through the gateway (port 8001).

══════════════════════════════════════════════════════════════════
SECTION 8 — WHAT EACH API RETURNS (Response Field Reference)
══════════════════════════════════════════════════════════════════

POST /api/auth/login → AuthResponse:
  accessToken, refreshToken, tokenType, expiresIn,
  userId, name, email, role

GET /api/users → UserResponse[]:
  userId, name, email, phone, role, isActive, createdAt, updatedAt

GET /api/services → ServiceResponse[]:
  serviceId, name, type, status, description,
  ownerUserId, configDetails, createdAt, updatedAt

GET /api/configurations/{serviceId} → ConfigurationResponse[]:
  configId, serviceId, parameter, value, effectiveDate,
  createdBy, createdAt

GET /api/usage/service/{serviceId} → UsageRecordResponse[]:
  usageId, serviceId, userId, metricType, value, unit, recordedAt

GET /api/sla → SlaDefinitionResponse[]:
  slaId, serviceId, metric, threshold, unit, operator,
  status, description, createdAt, updatedAt

GET /api/sla/breaches → SlaBreachEventResponse[]:
  breachId, slaId, serviceId, actualValue, thresholdValue,
  breachTime, resolved, resolvedAt

GET /api/incidents → IncidentResponse[]:

SECTION 9 — COMMON ISSUES & FIXES
══════════════════════════════════════════════════════════════════

ISSUE: Admin login returns 401 "Invalid credentials"
FIX  : Restart iam-service. AdminInitializer re-encodes password.
       Then retry: admin@comnet360.com / Admin@123

ISSUE: 400 on POST /api/configurations — effectiveDate error
FIX  : Must send { "parameter":"...", "value":"...", "effectiveDate":"YYYY-MM-DD" }
       effectiveDate is required, format must be ISO date string

ISSUE: 400 on POST /api/sla — unit error
FIX  : Must include "unit" field e.g. "unit": "ms"
       Also "metric" is free text not an enum

ISSUE: 400 on POST /api/reports/generate
FIX  : Must include ALL of: title, scope, reportType, fromDate, toDate
       reportType must be one of: OPERATIONS, SLA_COMPLIANCE,
       INCIDENT_SUMMARY, USAGE_ANALYTICS

ISSUE: POST /api/incidents/{id}/comments with {content:...} gets 400
FIX  : Use {"comment": "your text"} — field is "comment" not "content"

ISSUE: PUT /api/incidents/{id}/assign not working
FIX  : Use {"assignedTo": 3} — NOT {"userId": 3}
       Backend reads body.get("assignedTo")

ISSUE: Notification bell badge not updating
FIX  : Badge polls every 30 seconds. Wait or refresh the page.
       Check notification-service is running on port 8086.

ISSUE: Role-change notification not appearing
FIX  : Check iam-service logs for "✅ Notification sent" or "⚠️ Failed"
       Ensure notification-service is on port 8086
       Check application.yml: notification.service.url=http://localhost:8086

ISSUE: 503 Service Unavailable
FIX  : Service not registered in Eureka yet. Wait 30 seconds and retry.
       Check Eureka at http://localhost:8761 — service must appear.

ISSUE: Usage chart shows no data
FIX  : Record usage via POST /api/usage first (Steps 43-50)
       Then select the service from dropdown on Usage page

ISSUE: Incident detail shows "User #0" for reportedBy
FIX  : This happens when calling directly through Swagger (no gateway).
       Always test incidents through the gateway (port 8001) so the
       gateway injects X-User-Id header correctly.

ISSUE: Analytics getSlaCompliance returns 400
FIX  : All three params are required: serviceId, from, to
       Example: ?serviceId=1&from=2025-01-01&to=2025-12-31

ISSUE: Frontend shows "Service not found" after creating service
FIX  : The URL uses serviceId (from backend). If you see /services/undefined
       it means the old "id" field was being read. This was fixed in the
       latest frontend update. Hard refresh browser (Ctrl+F5).
