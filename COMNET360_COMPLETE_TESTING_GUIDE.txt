================================================================================
  TROUBLESHOOTING
================================================================================

"Cannot reach the server" on login
  → All 9 backend services must be running and showing UP in Eureka.
  → Eureka dashboard: http://localhost:8761

"Service unavailable" / 502 on login
  → IAM service not registered in Eureka. Restart IAM and wait 15 seconds.

"Invalid email or password"
  → Use Comnet@360 exactly (capital C, @, 360).
  → Wait 15–20 s after IAM start — AdminInitializer runs on boot.

"Assign dropdown is empty"
  → Only ADMIN, SERVICE_MANAGER, OPERATIONS_HEAD can fetch the users list.
  → Check IAM service running on port 8081.

"No SLA breach events appear after logging usage"
  → Breach fires instantly when usage is logged that violates the threshold.
  → Refresh the SLA Breach Events table immediately after logging usage.
  → SLAs must use ONLY: CALL_DURATION | MESSAGE_COUNT | VIDEO_MINUTES | BANDWIDTH
  → All other metric names are silently ignored.
  → Verify the SLA is ACTIVE (not INACTIVE/Disabled).

"Breach fires but I already resolved it — why a new one?"
  → A new breach is only created when new usage is logged that violates the threshold.
  → Resolve a breach → stay resolved until bad usage is logged again.
  → No automatic re-breach occurs on a timer.

"Minimum-usage rule not triggering"
  → Log a value clearly below the threshold (e.g. if the minimum is 600, log 280).
  → Breach fires the moment you click "Log Usage" — no waiting needed.

"Generate Report button not visible for a non-Admin user"
  → Only the ADMIN role can generate analytics reports.
  → SERVICE_MANAGER, OPERATIONS_HEAD, COMPLIANCE_OFFICER can VIEW reports but not generate.
  → This is by design — re-check hasRole('ADMIN') guard in AnalyticsPage.jsx.

"Add Action button not visible on Incident Detail"
  → Only ADMIN can add new resolution actions to incidents.
  → NETWORK_ENGINEER can UPDATE action status (PENDING → IN_PROGRESS → COMPLETED) only.
  → This is by design — re-check hasRole('ADMIN') guard in IncidentDetailPage.jsx.

"NE sees Create Incident button"
  → Network Engineers should NOT see "New Incident". Only ADMIN and SERVICE_MANAGER can create.
  → Check IncidentsPage.jsx — button guard should be hasRole('ADMIN', 'SERVICE_MANAGER').

"Notification says 'User #7' instead of user name"
  → IAM service must be running and the /api/users/{id}/name endpoint must be accessible.
  → Restart IAM service and incident-service / usage-sla-service.
  → Check SecurityConfig.java — "/api/users/*/name" must be in permitAll list.

"SLA breach notification title shows em-dash ( — ) or 'SLA #N'"
  → SlaScheduler.java and SlaService.java have been updated to use service names and colons.
  → Restart usage-sla-service to pick up changes.
  → Title format: "SLA Alert: <Service Name>" and "SLA Breach Resolved: <Service Name>".

"IN_PROGRESS badge missing after assignment"
  → Backend auto-sets IN_PROGRESS when an OPEN incident is assigned.
  → Refresh the page after assignment if the badge does not update instantly.

"Notifications not appearing after assignment"
  → Restart incident-service — NotificationClient must be compiled in.
  → Verify notification-service running on port 8086.
  → Check notification_db exists in MySQL.

"Chart shows no data after logging usage"
  → Select the correct service from the dropdown after logging.
  → Different metric types appear as separate data points on the same service.

"Landing page stats not animating"
  → Ensure JavaScript is enabled in the browser.
  → Hard-refresh (Ctrl+Shift+R) to clear any cached build.

"TopBar shows Offline instead of Live"
  → The real-time polling hook failed to fetch from the backend.
  → Ensure all backend services are running. Refresh the browser.

"Profile page avatar is blue instead of role-specific colour"
  → The avatar gradient is driven by user.role. Ensure the user was created
    with the correct role in Phase 1. Re-check /admin/users to confirm.

"Redirected to landing page instead of login when accessing /dashboard"
  → This is CORRECT behaviour. Unauthenticated users are redirected to /
    (the landing page), not to /login.
