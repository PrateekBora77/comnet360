PART 1 — WHAT WAS FIXED AND WHY
────────────────────────────────────────────────────────────

1. ADMIN LOGIN (iam-service)
   Problem : DB had a broken BCrypt hash seed from original SQL dump.
   Fix     : AdminInitializer.java (ApplicationRunner) re-encodes the
             admin password on every startup using Spring's own encoder.
   File    : comnet360/services/iam-service/src/main/java/com/comnet360/iam/config/AdminInitializer.java
   Login   : admin@comnet360.com / Admin@123

2. ADMIN UPDATE USER ROLE/STATUS
   Problem : Frontend sent /api/users/undefined/role because User type
             used "id" but backend returns "userId".
   Fix     : Updated User type, AuthContext, UsersPage to use "userId"
             and "isActive: boolean" (not "status: string").

3. ROLE-CHANGE NOTIFICATIONS
   Problem : NotificationClient used @LoadBalanced RestTemplate with
             Eureka service name — failed silently when Eureka not synced.
   Fix     : Plain RestTemplate with direct URL http://localhost:8086.
   Files   : NotificationClient.java, RestTemplateConfig.java, application.yml

4. NOTIFICATIONS PAGE / TOP BAR
   Problem : Notification type used wrong field names (id, read, createdAt).
             Backend returns: notificationId, status (UNREAD/READ), createdDate.
   Fix     : Updated Notification type and NotificationsPage, TopBar.

5. ALL OTHER ENTITY FIELD MISMATCHES (see Part 2)
   Every service had mismatches between backend DTO field names and
   what the frontend was using. All fixed in this session.

PART 2 — FIELD NAME MAPPING (Backend → Frontend)
────────────────────────────────────────────────────────────

PROVISIONING SERVICE (port 8082)
  ServiceResponse:
    serviceId       (was: id)
    ownerUserId     (was: ownerId / ownerEmail — no email field exists)
    name, description, type, status, createdAt, updatedAt ✓

  ConfigurationResponse:
    configId        (was: id)
    parameter       (was: key)
    effectiveDate   (was: description — date string not description text)
    serviceId, value, createdAt ✓
    NOTE: CreateConfigurationRequest needs { parameter, value, effectiveDate }
          effectiveDate is @NotNull — must be ISO date "YYYY-MM-DD"

USAGE-SLA SERVICE (port 8083)
  UsageRecordResponse:
    usageId         (was: id)
    metricType, serviceId, userId, value, unit, recordedAt ✓

  SlaDefinitionResponse:
    slaId           (was: id)
    metric: String  (was: metricType — it's a plain String, not an enum)
    unit: String    (NEW field — required in CreateSlaRequest)
    threshold: BigDecimal, operator, status, description, createdAt ✓
    NOTE: CreateSlaRequest needs { serviceId, metric, threshold, unit, operator, description }
          "metric" is a free-text String, not the MetricType enum

  SlaBreachEventResponse:
    breachId        (was: id)
    thresholdValue  (was: threshold)
    breachTime      (was: breachedAt)
    resolved, resolvedAt, actualValue, slaId, serviceId ✓
    NOTE: No "metricType" field on breach — only slaId reference

INCIDENT SERVICE (port 8084)
  IncidentResponse:
    incidentId      (was: id)
    detectedDate    (was: createdAt)
    resolvedDate    (was: resolvedAt)
    reportedBy      (was: reportedById/reportedByEmail — just a Long userId)
    assignedTo      (was: assignedToId/assignedToEmail — just a Long userId or null)
    NOTE: No serviceName field — only serviceId
    NOTE: PUT /incidents/{id}/assign body must be { assignedTo: userId }
          NOT { userId: userId }

  IncidentCommentResponse:
    commentId       (was: id)
    comment         (was: content)
    authorId        (was: authorEmail — just a Long userId)
    incidentId, createdAt ✓
    NOTE: AddCommentRequest body must be { comment } NOT { content }

  ResolutionActionResponse:
    actionId        (was: id)
    actionDescription (was: title/description — single field, no title)
    ownerId         (was: ownerEmail — just a Long userId)
    dueDate         (NEW — optional LocalDateTime)
    status, completedAt, createdAt ✓
    NOTE: CreateActionRequest body must be { ownerId, actionDescription, dueDate? }
          There is NO "title" field

ANALYTICS SERVICE (port 8085)
  KpiReportResponse:
    reportId        (was: id)
    reportType      (was: type)
    generatedBy     (was: generatedByEmail — just a Long userId)
    generatedDate   (was: createdAt)
    scope           (NEW field)
    metrics         (was: data — it's a String, not Record<string,unknown>)
    fromDate, toDate ✓
    NOTE: GenerateReportRequest needs { title, scope, reportType, fromDate, toDate }
          ALL fields are @NotNull/@NotBlank — none are optional
          "reportType" must be one of: OPERATIONS, SLA_COMPLIANCE,
          INCIDENT_SUMMARY, USAGE_ANALYTICS

  DashboardSnapshotResponse:
    snapshotId      (was: id)
    snapshotType    (was: type)
    data: String    (JSON string — not Record<string,unknown>)
    capturedAt ✓

  GET /reports/sla-compliance  — requires serviceId, from, to (@RequestParam, all required)
  GET /reports/incident-summary — requires from, to (@RequestParam, both required)

NOTIFICATION SERVICE (port 8086)
  NotificationResponse:
    notificationId  (was: id)
    status: UNREAD|READ  (was: read: boolean)
    createdDate     (was: createdAt)
    userId, title, message, category, channel ✓

  NotificationPreferenceResponse:
    prefId, emailEnabled, inAppEnabled, categories: String ✓
    (no incidentAlerts/slaAlerts/serviceUpdates — removed in backend)

PART 8 — COMMON ERRORS AND FIXES
────────────────────────────────────────────────────────────

"Invalid credentials" on login:
  → Restart iam-service. AdminInitializer will re-sync the password.

404 on /api/... endpoints:
  → Check api-gateway is running on port 8001.
  → Check frontend .env: VITE_API_URL=http://localhost:8001/api

400 Bad Request on Create Configuration:
  → Must send: { "parameter": "...", "value": "...", "effectiveDate": "YYYY-MM-DD" }
  → effectiveDate is @NotNull — cannot be empty

400 Bad Request on Generate Report:
  → Must send all 5 fields: title, scope, reportType, fromDate, toDate
  → reportType must match enum: OPERATIONS, SLA_COMPLIANCE,
    INCIDENT_SUMMARY, USAGE_ANALYTICS

400 Bad Request on Create SLA:
  → Must include "unit" field (e.g. "Mbps", "ms", "%")
  → "metric" is free text, not the MetricType enum

Notification not received after role change:
  → Check notification-service is running on port 8086.
  → Check application.yml: notification.service.url=http://localhost:8086
  → Check iam-service logs for "✅ Notification sent" or "⚠️ Failed"

SLA compliance chart not loading:
  → getSlaCompliance requires serviceId + from + to — all required params.
  → Select a service and date range before calling the API.

Incident detail shows "User #X" instead of email:
  → This is correct — backend IncidentResponse only returns userId (Long),
    not email. The email was never stored in the incident record.
